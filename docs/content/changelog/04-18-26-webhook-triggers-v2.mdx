Webhook Triggers V2 introduces a first-class `webhook_endpoints` resource with a dedicated ingress URL per OAuth app. V2 is **opt-in** and scoped to new trigger slugs — existing V1 triggers, URLs, and payload formats are unchanged.

## Summary

| Change | Action required |
|---|---|
| New `webhook_endpoints` API | None (opt-in) |
| New ingress path `/api/v3/webhook_ingress/{toolkit}/{we_*}/trigger_event` | None (opt-in) |
| Ingress-level signature verification | Automatic for V2 endpoints |
| New Slack V2 slugs (`SLACK_CHANNEL_MESSAGE_RECEIVED`, `SLACK_DIRECT_MESSAGE_RECEIVED`) | Opt-in |
| V1 ingress path and all legacy trigger slugs | None — unchanged |

## What's in V2

**Dedicated endpoint per OAuth app.** Each `webhook_endpoint` is keyed by `(toolkit_slug, project_id, client_id)` and exposes its own URL containing a random `we_*` identifier:

```
https://backend.composio.dev/api/v3/webhook_ingress/{toolkit}/{we_xxx}/trigger_event
```

This replaces V1's shared `/handle` URL per toolkit. Events arriving on a V2 URL are fanned out only to trigger instances on that endpoint's project.

**Signature verification at ingress.** Apollo cryptographically verifies every V2 request against the signing secret stored on the endpoint — HMAC-SHA256, Ed25519, or `token_match` depending on the toolkit. Unsigned or tampered payloads are rejected, so third parties cannot spoof events onto your triggers. For providers that sign a request timestamp (e.g. Slack), replay protection additionally rejects requests whose timestamp falls outside the allowed skew window.

**Per-user authorization for app-level events.** Toolkits with per-user visibility — Slack being the first — can use an app-level token (e.g. `xapp-…` with `authorizations:read`) to resolve which connected users are authorized for a given event. Only those users' triggers fire. This enables safe delivery of private channel and DM events that V1 silently dropped.

**Config-level filters.** V2 triggers filter events against `trigger_config` at match time:

```json
{
  "trigger_config": {
    "channel_id": "C_SUPPORT",
    "is_bot_message": "false",
    "is_thread_reply": "false"
  }
}
```

**Lifecycle support.** Provider handshakes (Slack `url_verification`, Asana `X-Hook-Secret`, Outlook validation tokens) are handled at ingress. Provider-side webhooks are deregistered when user-level triggers are deleted.

## API reference

All endpoints require a project API key (`x-api-key`).

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/v3/webhook_endpoints/schema?toolkit_slug={slug}` | List required setup fields for a toolkit |
| `POST` | `/api/v3/webhook_endpoints` | Create an endpoint (idempotent per `toolkit_slug + client_id` within a project) |
| `GET` | `/api/v3/webhook_endpoints` | List endpoints in the current project |
| `GET` | `/api/v3/webhook_endpoints/{id}` | Get a single endpoint, including `verified_at` |
| `POST` | `/api/v3/webhook_endpoints/{id}` | Replace `setup_data` |
| `PATCH` | `/api/v3/webhook_endpoints/{id}` | Update one or more fields in `setup_data` |
| `DELETE` | `/api/v3/webhook_endpoints/{id}` | Soft-delete |

Provider events are posted to:

| Method | Path |
|---|---|
| `POST` | `/api/v3/webhook_ingress/{toolkit_slug}/{webhook_endpoint_nano_id}/trigger_event` |

## Migration walkthrough (Slack)

Slack is the first toolkit on V2. The same flow applies to every toolkit added later.

### 1. Discover required fields

```bash
curl "https://backend.composio.dev/api/v3/webhook_endpoints/schema?toolkit_slug=slack" \
  -H "x-api-key: YOUR_API_KEY"
```

### 2. Create the endpoint

```bash
curl -X POST "https://backend.composio.dev/api/v3/webhook_endpoints" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "toolkit_slug": "slack", "client_id": "YOUR_SLACK_CLIENT_ID" }'
```

The response includes `webhook_url`. The call is idempotent.

### 3. Store the signing secret

```bash
curl -X PATCH "https://backend.composio.dev/api/v3/webhook_endpoints/ENDPOINT_ID" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "data": { "webhook_signing_secret": "YOUR_SIGNING_SECRET" } }'
```

<Callout type="warn">
Store the signing secret before switching the provider's callback URL. If Slack posts to a V2 URL without a secret available, every request fails with `400` and Slack may auto-disable the endpoint after ~36 hours of failures.
</Callout>

### 4. Point your Slack app at the V2 URL

Set **Event Subscriptions → Request URL** in your Slack app to the `webhook_url` returned in step 2. Slack issues a `url_verification` challenge; Apollo responds and sets `verified_at` on the endpoint.

### 5. (Optional) Add an app-level token for private channel and DM triggers

Required only for triggers that need per-user authorization.

```bash
curl -X PATCH "https://backend.composio.dev/api/v3/webhook_endpoints/ENDPOINT_ID" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "data": { "app_token": "xapp-..." } }'
```

Generate the token from **Slack app → Basic Information → App-Level Tokens** with scope `authorizations:read`.

### 6. Create V2 trigger instances

| V1 slug (still supported) | V2 replacement | Notes |
|---|---|---|
| `SLACK_RECEIVE_MESSAGE` | `SLACK_CHANNEL_MESSAGE_RECEIVED` | Adds `channel_id`, `is_bot_message`, `is_thread_reply` filters |
| `SLACK_RECEIVE_DIRECT_MESSAGE` | `SLACK_DIRECT_MESSAGE_RECEIVED` | Requires `app_token` for per-user authorization |

```bash
curl -X POST "https://backend.composio.dev/api/v3/trigger_instances/SLACK_CHANNEL_MESSAGE_RECEIVED/upsert" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "connected_account_id": "CONNECTED_ACCOUNT_ID",
    "trigger_config": { "channel_id": "C_SUPPORT", "is_bot_message": "false" }
  }'
```

Creating a V2-only slug without a matching `webhook_endpoint` returns `400` with a `suggestedFix` pointing to `POST /api/v3/webhook_endpoints`.

## Constraint: one OAuth app, one project

`webhook_endpoints` uniqueness is `(toolkit_slug, project_id, client_id)`. More importantly, the provider dashboard accepts exactly **one callback URL per OAuth app**, and each V2 URL resolves to a single project.

<Callout type="warn">
An OAuth app (`client_id`) on V2 can effectively serve only one Composio project. If the same OAuth app is currently shared across multiple projects or organizations on V1, either pick one project to own it on V2 or register separate OAuth apps per project.
</Callout>

Scoping V2 endpoints to a single project gives each project:

- Its own ingress rate-limit and backpressure budget
- Isolated fan-out — events reach only that project's trigger instances
- Isolated signing secret and app-level token storage
- Accurate per-project metering and billing
- Independent teardown, rotation, and verification state

V1's shared endpoint continues to work for any OAuth app left on V1, with no change in behavior.

## Backward compatibility

- V1 ingress (`/api/v3/trigger_instances/{toolkit}/{project_id}/handle`) is **not deprecated**.
- All existing trigger slugs — every `SLACK_RECEIVE_*`, `SLACKBOT_*`, and equivalent legacy slug across toolkits — continue to route through V1 unchanged.
- No server-side data migration touches existing trigger instances.
- V1, V2, and V3 outbound webhook payload formats are unaffected.

## Availability

V2 is opt-in. Slack is the first toolkit available; HubSpot, Linear, and other app-level toolkits will follow the same pattern and be announced in their own changelog entries. There is no migration deadline — adopt V2 when you need its features.
